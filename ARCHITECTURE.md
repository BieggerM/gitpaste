# Architecture and security decisions

The shell hook is deliberately a narrow dispatcher, not the security boundary. Bash calls `command_not_found_handle` and Zsh calls `command_not_found_handler` only after normal command lookup fails. Each hook requires exactly one argument, applies a cheap anchored pattern for a GitHub/GitLab HTTPS or SCP-style SSH URL, and passes the original value as one quoted argument to `gitpaste clone --`. A non-candidate returns the conventional status 127 or delegates to a safely captured prior handler.

Zsh exposes function bodies through its `functions` associative array, allowing an existing handler to be copied and chained without evaluating generated code. Bash offers no equivalent safe reference or rename operation. The Bash integration therefore refuses to replace an existing handler and explains the limitation. It does not use `eval` or source generated function definitions.

`internal/repositoryurl` is the authoritative boundary. It uses anchored ASCII allowlist patterns plus structured HTTPS parsing. Only `github.com` and `gitlab.com` are accepted. HTTPS is the only URL scheme; SSH uses the conventional `git@host:path` form. Credentials, ports, percent escapes, queries, fragments, whitespace, control characters, metacharacters, option-like values, traversal segments, and unexpected path depth are rejected.

`internal/clone` validates again immediately before execution. It invokes `git` directly with `exec.CommandContext` and the argument vector `clone`, `--`, and the unchanged validated URL. Standard streams are attached directly. No command shell, evaluation, or interpolated command string exists in the execution path. A Git child exit status is returned unchanged; failure to locate or start Git is reported clearly with status 127.

The installer writes fixed, embedded hook assets to an owner-only user-local config directory and requires absolute paths. Hook-file and config-directory symlinks are rejected; symlinked shell rc files are deliberately resolved so dotfile-manager links remain intact. New contents are written to a permission-controlled temporary file, synced, and atomically renamed. Shell rc modifications are bounded by shell-specific start/end markers. Installation is idempotent, and uninstall rejects malformed marker blocks rather than risk deleting unrelated configuration.

Cloning only transfers repository data; gitpaste never runs code from the checkout. That does not make a repository trustworthy. Users must inspect cloned content before running its scripts, builds, or installation instructions.

The command-not-found API runs after the interactive shell has parsed its input. Consequently, no hook can inspect or neutralize shell syntax outside the command word it receives. For example, if a user pastes an unquoted semicolon followed by another command, the shell parses that second command before gitpaste is involved. The validator rejects metacharacters whenever they are part of the value passed to it, but gitpaste cannot protect users from pasting arbitrary shell programs. The integration is intended for repository URLs copied from a trusted UI as a single command word.
